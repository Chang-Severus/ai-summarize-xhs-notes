## Agentic RL（智能体强化学习）

### 1. Agentic RL 与 Reasoning RL 的本质差异：从单轮封闭计算变成多轮分布式系统问题 `高价值`
Reasoning RL（数学/代码）是封闭循环：出题→答题→对答案→算梯度，奖励就是“对不对”。而 agent RL 需调 API、读库、与模拟用户多轮交互，一个 episode 可能十几轮，每轮等外部环境返回，本质是分布式系统问题：要同时管理 GPU 推理、外部环境异步调用、组件间数据流。

- [agentic rl算法侧](https://www.xiaohongshu.com/explore/69f22c620000000022025283?xsec_token=ABrHBu_UR-t3UsOyN9OL-yi9lHeMaEnp3i51CfMNZHkSs=&xsec_source=pc_user)

### 2. 蚂蚁开源 AReaL 的关键设计：把训练系统伪装成 OpenAI 兼容 API，抹掉“训练”和“运行”的边界 `高价值`
Agent 代码照常调 chat completion API，调试时接真 API，训练时只把 base_url 换成 AReaL 地址，代码一行不改。这让 agent 开发者不需懂 RL、RL 研究者不需懂 agent 框架，两边解耦。另配合全异步训练（生成与训练拆到不同 GPU 集群、replay buffer 解耦），实测 2.77 倍加速且性能不降。

来源：
- [怎么让Agent 的 RL训练真正跑起来](https://www.xiaohongshu.com/explore/69a91ed1000000001b01726e?xsec_token=ABs5KkqDt5HjpEjLlaG3OkJQgVRL-sniQ64eDczx_kiFA=&xsec_source=pc_user)

### 3. 异步 RL 的算法硬伤：旧权重生成的“脏数据”破坏 PPO 的同策略假设，需用 staleness 上界 + decoupled PPO 约束 `高价值`
生成端用旧权重，训练端拿到的数据可能来自好几个版本前的 policy；甚至一条 trajectory 前半段是 v3、后半段是 v5 生成的。标准 PPO 假设所有 action 来自同一 policy，该假设被打破。解法是引入 staleness 上界和 decoupled PPO objective 用数学约束住。

来源：
- [怎么让Agent 的 RL训练真正跑起来](https://www.xiaohongshu.com/explore/69a91ed1000000001b01726e?xsec_token=ABs5KkqDt5HjpEjLlaG3OkJQgVRL-sniQ64eDczx_kiFA=&xsec_source=pc_user)

### 4. Agentic RL 信用分配（Credit Assignment）的核心：把 episode 的最终成败拆解到每一步中间决策上 `高价值`
多轮轨迹（读任务→搜索→读证据→调工具→总结）通常只在最后给 R_final=0/1。难点是把 trajectory-level 标量回拆成每步的 step-level advantage（A_t），再乘到对应 decision tokens 的 log-prob 上更新；observation（工具/网页返回）通常被 mask。若只做 trajectory-level RL，失败轨迹里的好搜索会被一起惩罚，成功轨迹里的废操作会被一起奖励。

- [agentic rl算法侧](https://www.xiaohongshu.com/explore/69f22c620000000022025283?xsec_token=ABrHBu_UR-t3UsOyN9OL-yi9lHeMaEnp3i51CfMNZHkSs=&xsec_source=pc_user)

### 5. GiGPO 用“跨 episode 的相同 anchor state”聚合动作，低成本实现 step 级信用分配 `高价值`
Vanilla GRPO 只给整个 episode 一个 reward，学不到每步信号；直接对每个 step 多次 rollout 计算开销过高。GiGPO 发现不同 episode 中模型常面临相同状态（anchor state），把这些跨 episode 的重复状态作为一个 rollout batch 提供相对优势信号，episode level 优势用 Dr.GRPO 的估计方法。

来源：

### 6. IGPO：用“每轮交互是否让模型更易生成正确答案”作为稠密奖励信号 `高价值`
若某轮交互让模型更容易生成正确答案就该奖励，偏离则惩罚。优点是便宜、稠密、ground-truth-aware，不需额外 PRM 或蒙特卡洛估值；缺点是依赖高质量 ground truth，问题有多个合理答案而数据集只认一个时，可能惩罚事实正确但不匹配标注的路径。

### 7. Agent 多轮自进化为何会坍塌：原则级经验 + 步进式注入 + 离策略蒸馏是最优组合 `高价值`
系统研究经验内化三维度，发现 on-policy context-distillation 会导致多轮坍塌；步进式注入可把“过早回答”从 63.82% 降到 0%（+8.0pp），off-policy 蒸馏提供连贯行为示范，实现跨迭代稳定自进化（3 轮持续提升）。论文 arxiv 2606.04703。

### 8. AEnvironment：把训练/测评/部署所需的环境统一抽象成 MCP 协议接口 `一般`
训练、测评、部署 agent 都需要环境，每次从零搭门槛太高。AEnvironment 把一切抽象成 MCP 的 Environment 接口——bash 终端、benchmark、甚至另一个 agent 都可以是环境，用同一套接口注册工具、定义奖励、跑评测，从训练到上线不换接口。

## RL 训练实战技巧与稳定性

### 1. KL loss 用 k3 而非 k1：k1 会持续惩罚本就小的 log_p 导致难任务上模型坍塌 `高价值`
用 slime 训 computer use agent 总训一半崩，元凶是默认用了 k1 KL loss。k1 的缺陷是持续惩罚本来就小的 log p，难 task 上 log p 本就要被 reward 惩罚、再被 k1 惩罚模型立即坍塌；k3 保证 KL loss 严格为正，完美解决。

来源：
- [做了一年RL的小经验](https://www.xiaohongshu.com/explore/69adb8c8000000001a01c6fd?xsec_token=ABI_Z_LYdXCXGP0IKIF3xenDB7-Ug9SZPuI6tjC_A1MI8=&xsec_source=pc_user)

### 2. value model 并非多余：它赋予 token level advantage，能很好稳定训练，业界主流仍用 PPO `高价值`
很多人受各种 g*po 影响认为 value model 多余，但实践中 value model 提供 token level advantage 对稳定训练很重要，业界大多还是用 PPO。

来源：
- [做了一年RL的小经验](https://www.xiaohongshu.com/explore/69adb8c8000000001a01c6fd?xsec_token=ABI_Z_LYdXCXGP0IKIF3xenDB7-Ug9SZPuI6tjC_A1MI8=&xsec_source=pc_user)

### 3. RL 常见超参经验：大 batch RL loss 默认 1e-6，调成 SFT/RLHF 的 1e-5 通常会崩 `高价值`
updates per step 一般设 1 保证尽量 on-policy；rollout per step（task×sample）越多越稳定；学习率默认 1e-6，新手为加速调到 1e-5 左右通常很寄。还需关注 task 难度、clip ratio、kl coefficient、temperature 是否合理。

来源：
- [做了一年RL的小经验](https://www.xiaohongshu.com/explore/69adb8c8000000001a01c6fd?xsec_token=ABI_Z_LYdXCXGP0IKIF3xenDB7-Ug9SZPuI6tjC_A1MI8=&xsec_source=pc_user)

### 4. agentic RL 要做好 advantage standardization，并清洗被低质量 env feedback 污染的样本 `高价值`
每个 step 的 advantage 估计若不做好 standardization 会直接影响效果；真实复杂 env 的不稳定性会污染样本，收集训练数据前要去掉未完成或被低质量 env feedback 污染的样本。为防长尾问题普遍用异步，这种 off-policy 从实验看影响不大。

来源：
- [做了一年RL的小经验](https://www.xiaohongshu.com/explore/69adb8c8000000001a01c6fd?xsec_token=ABI_Z_LYdXCXGP0IKIF3xenDB7-Ug9SZPuI6tjC_A1MI8=&xsec_source=pc_user)

### 5. KL 放 Reward 还是 Loss：决定的不是记号，而是 advantage 语义和梯度通路 `高价值`
PPO-RLHF 通常 KL in reward（KL 当作样本成本，先改 reward 再改 return/GAE/advantage）；GRPO 通常 KL in loss（group advantage 表达 raw reward 相对好坏，KL 作独立 actor regularizer）；DAPO/Dr.GRPO 等 long-CoT recipe 开始质疑 reference KL 本身，no KL 不是免费午餐，稳定性压力转给 clip、reward design、sampling 和监控。读代码判断：rewards 含 non_score_reward→reward-side；有独立 kl_loss→loss-side。

### 6. 业界 agentic RL 算法普遍采用最简单的 GRPO 形式，非科研可直接参考 `一般`
相关论文大多比较 tricky，但业界普遍用最简单的 GRPO：在整个 episode 上给一个 reward，训练时多轮直接 concat，不区分不同 step。如果不是为了科研，直接参考 GRPO 方案即可。

## 后训练：SFT / RL / 蒸馏（OPD）

### 1. SFT、RL、OPD 本质都是在重塑模型输出分布，区别只在“目标分布是什么、数据从哪来” `高价值`
一篇 blog（nrehiew）的核心 framing：SFT 遗忘多是因为目标分布外部固定、离多远硬拉多远；RL 遗忘少是因为数据从自身采样、隐式锚定在原始策略附近；OPD 继承 RL 的抗遗忘是因为同样 on-policy sampling。三者区别归根到底是“数据从谁的分布里来”这一变量，on-policy data 是核心。

来源：
- [最近最喜欢的一篇blog：关于post-training](https://www.xiaohongshu.com/explore/6a2e6a78000000002101b784?xsec_token=ABxSpNAJIZ6n94O5sVBon_MtpDB0QTdsUfj0vufRD-6h0=&xsec_source=pc_user)

### 2. 反直觉实验：从“遗忘严重”的 SFT teacher 蒸馏出的 student 反而不怎么遗忘 `高价值`
用 SFT 和 RL 各训一个 teacher（RL teacher 表现好不遗忘，SFT teacher 学会任务但遗忘严重），再用 On-Policy Distillation 让 student 分别跟两个 teacher 学，两个 student 最终几乎一样好且都超过 RL teacher。原因是 OPD 数据由 student 自己采样，teacher 只提供方向，student 每步都锚定在自己分布附近。

来源：
- [最近最喜欢的一篇blog：关于post-training](https://www.xiaohongshu.com/explore/6a2e6a78000000002101b784?xsec_token=ABxSpNAJIZ6n94O5sVBon_MtpDB0QTdsUfj0vufRD-6h0=&xsec_source=pc_user)

### 3. OPD（On-Policy Distillation）= on-policy 数据采集 + token-level 蒸馏 `高价值`
让 student 先按当前策略生成 prefix，teacher 只在 student 真实走到的 prefix 上给 token-level 监督（不生成完整答案）。loss 选择遵循“先问 prefix 是否 on-policy，再问 teacher 是否给 token distribution，最后才问 KL 方向”：多解/开放生成用 FKL/JSD，数学/可验证推理用 RKL/advantage，teacher 高熵 token 多用 EOPD，teacher-student gap 大用 JSD/GKD 插值。

来源：
- [OPD：大模型后训练中的在线策略蒸馏](https://www.xiaohongshu.com/explore/6a2f652b000000000f0170a3?xsec_token=ABmMFWR9z4k4ztyjRZv8SUF7jR8l1Am7IRL6P2gKbHlEc=&xsec_source=pc_user)

### 4. 如何判断 SFT 到什么程度可以开始做 RL：三个信号 `高价值`
SFT 建立基础能力（格式、指令遵循），RL 优化输出质量，两者不能互相替代；过早做 RL 会训练崩溃（低奖励陷阱、策略崩塌、KL 散度爆炸），过晚浪费算力。可停止 SFT 的信号：①格式和能力已对齐（指令理解准确率>85%）；②性能达上限（验证集 Loss 稳定、关键指标多轮不再提升、继续加数据过拟合）；③pass@k 明显高于 pass@1（说明模型知道答案形态但输出不稳定，RL 能把“偶尔正确”变“稳定正确”）。

来源：
- [如何判断SFT到什么程度就可以开始做RL了？](https://www.xiaohongshu.com/explore/69dde2190000000023025763?xsec_token=ABBArpNrbzftITf6iNmgieC2UrWQmVz5VNxqxQC5zfotA=&xsec_source=pc_user)

## 推理优化与高性能算子

### 1. Batch Size 越大显卡越“卖力”的本质：把访存 Bound 转成算力 Bound `高价值`
AI 芯片“重算轻搬”：HBM3 卡算力≈2000 TFLOPS、带宽≈4.8 TB/s，机器算术强度≈416 FLOPs/Byte，即每搬 1 字节要算 416 次以上才不饿肚子。Batch=1 时算术强度仅 2，99% 时间在等数据（访存 Bound）；Batch=N 时权重驻留缓存复用 N 次，算术强度线性飙升，跨越 416 临界点进入算力 Bound。大 Batch 提吞吐但增加单请求延迟，这是用空间换时间，也是 vLLM 等卷“连续批处理”的原因。

来源：
- [为什么Batch Size越大，显卡跑得越“卖力”](https://www.xiaohongshu.com/explore/6a321829000000000f00597f?xsec_token=ABj86SkxO9_n7dyYu_MDy2Am_OZ5I_akeCziH2kvTO2jk=&xsec_source=pc_user)

### 2. 手写 GPU 算子的技术选型：永远先 torch.compile，不足再上 Triton，最后才手写 CUDA `高价值`
CS336 第六讲以 GeLU 为例：CUDA C++ 最底层、开发调试成本极高；Triton 把抽象从线程提升到线程块、性能与 CUDA 持平但效率高；torch.compile 一行代码自动融合、GeLU 案例最快(1.47ms)。非融合与融合实现性能差距可达近 8 倍。核函数融合（Kernel Fusion）把多层计算打包成一个计算图减少内存带宽压力。

来源：
- [CS336 Lecture 6 笔记：手写高性能算子实战](https://www.xiaohongshu.com/explore/6a26e02b000000002202b584?xsec_token=AB8lZA9AvtPNCQ4uCqHwEEruzdvEEYEVjEGL9W2h_VPzI=&xsec_source=pc_user)

### 3. GPU 基准测试的两个易错点：Warm-up 与 Synchronization `高价值`
①Warm-up：第一次跑 PyTorch 有编译机器码、发指令、分配内存池等一次性开销，不预热测的是含启动开销的速度；②Synchronization：CPU/GPU 异步，CPU 提交任务后立即往下走，不调 torch.cuda.synchronize() 会在 GPU 还在工作时就记录结束时间，测到的时间无限接近零。永远先测量定位瓶颈再优化，且优化有边际效应（8.1ms→1.1ms 值得，1.1ms→1.0ms 可能不值）。

### 4. 混元 Hunyuan 3.0（285B MoE）推理优化实战：算子融合 + 多级 KV Cache + 量化 `高价值`
H20 上：rope/norm/量化/写 KV cache 全融成一个只读一次数据→5x；采样小 kernel（Top-K/Top-P/温度/softmax）从十几个压成两个→2.5~3.2x；Router GEMM 用两个 BF16 拼 FP32 精度跑 Tensor Core→约 3x；三级 KV Cache(HBM→DRAM→本地盘)命中省约 10% 计算；W4A8 量化精度差<1% 而吞吐+28%。Blackwell 上 weight offloading 把 285B 模型塞进一张卡，H2D/D2H 与计算 overlap，QPM 最高+28%。

## 模型结构与位置编码

### 1. RoPE 的“快慢针”设计哲学：高频特征像显微镜、低频特征像望远镜 `高价值`
前几对特征转速极快（高频）：能精准区分相邻极近的词，但转几圈就“套圈”(Aliasing)、距离一长就迷路；最后几对特征转速极慢（低频）：能处理长距离宏观夹角不套圈，但对相邻词几乎没反应。类似汽车里程表的进位制：个位跳动快、万位半天不动。RoPE 把 128 个特征分工（前 1/3 抓语法、中 1/3 抓逻辑、后 1/3 抓长文档定位）；Llama-3 调大 theta 本质是加更慢的“世纪指针”撑起 128k 视野。

## Agent 工程与评估（Eval）

### 1. Agent Eval 是近中期最实在的技术护城河：eval 是你的梯度，沉淀的 ground truth 才值钱 `高价值`
底座模型差不多，拉开差距的是谁能把“好/坏”定义得更准。值钱的不是 eval 框架（开源一抓一把），而是“在我业务里什么叫做对了”这套攒出来的 ground truth。三层评估：①单条轨迹打分——能用代码判的(格式/工具调用/数字)绝不交给模型，剩下才用 LLM as judge，且 judge 本身也得先被人工金标准对齐；②群体聚类；③闭环加速器。要评的是整条轨迹（怎么想/调什么工具/如何决策），不是最后那段文本——最终答案对但轨迹是定时炸弹的 case 很常见。

## AI 编程工作流（Vibe Coding）

### 1. Vibe Coding 最重要的技能是 Git，不是提示词：AI 只有上下文没有版本记忆 `高价值`

## 微调入门

### 1. 单卡入门微调首选 LoRA：只训低秩矩阵，参数量降到千分之一 `高价值`
LoRA 不改原模型参数、只额外训练低秩矩阵，24G 显存可跑 7B 模型两小时见效。数据用指令-响应对(JSON/CSV)，几百到几千条即可，质量比数量重要。模型从 7B 的 chat 版本起步(如 Qwen2.5-7B-Instruct，不要 base)，显存不够上 4bit 量化(QLoRA 可把 7B 压到约 11G)。用 HuggingFace Transformers+PEFT，学习率 1e-5~1e-4、训练 3~5 轮(太多过拟合)。注意灾难性遗忘：混入通用问答数据可缓解。
